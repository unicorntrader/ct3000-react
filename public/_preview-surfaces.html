Remaining inconsistencies across surfaces

  • Exit price source of truth. DailyView still uses buildExitInfo (raw-trade scan). Everyone else now reads avg_exit_price from DB. ShareModal reads row.exit passed in from DailyView, so inherits the indirection.
  • Journal's "Return %" column. Shows — for orphans because % needs entry; that's correct behaviour. But the % is computed from native P&L + cost basis, not (exit-entry)/entry — subtle inconsistency with ShareModal's Return % formula.
  • Daily View still uses buildExitInfo. Fine for normal closed trades but it's a second implementation path that can drift from the builder.
